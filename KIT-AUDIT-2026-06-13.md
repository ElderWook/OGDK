# OGDK — Independent Audit & Innovation Review

**Date:** 2026-06-13 · **Scope:** the kit itself (process, tooling, onboarding), audited
against one stated goal: *give this to someone with great ideas but no programming
experience, let them ship a first real project at surprising speed, and have them come
out the other side actually understanding how software is built.*

Method: read every top-level doc, every `tools/` script and both hooks, all five skills,
the app exemplar and architect, and the lessons/roadmap. Ran `gate.sh` on a native copy
(not the mount) — **GATE PASSED, exit 0**, all checks green. Ran a separate twin-parity
pass across all 16 `.ps1`/`.sh` pairs.

---

## 1. Verdict

This is not a starter template; it's a **discipline encoded as machinery**, and it is far
above the bar for a one-person operation. The learning loop (friction → `LESSONS.md` →
`kit-retro` → gate that verifies it never recurs), the *recording of declined decisions so
they aren't re-litigated*, the EOF-sentinel truncation guard, the byte-verified fleet
propagation with provenance stamps, the index-only privacy reporting — these are practices
most professional teams don't have. The kit is honest about its own state and visibly
shaped by real scars rather than speculation.

**But there is one structural problem that sits directly across the stated goal:** the kit
is written *by* and *for* the expert who built it, not yet for the beginner it wants to
serve. Almost every artifact front-loads paranoia (NTFS truncation, MSYS2 PATH poisoning,
private-marker scans, fleet propagation, history rewrites) that a first-time builder
neither needs nor can absorb on day one. The guardrails are excellent; the **on-ramp to
them is a wall**. The highest-leverage work is not adding more capability — it's *hiding*
most of the existing capability until the moment it's needed, and turning the parts a
beginner does touch into teaching rather than ritual.

If I had to compress it: **the kit is a 9/10 maintenance organism and a 4/10 gift to a
non-programmer.** Closing that gap is what this review is about.

---

## 2. What is genuinely excellent (keep, protect, don't dilute)

- **The learning loop with declined-decision memory.** `LESSONS.md` → `kit-retro` →
  mechanical gate is a real ADR/retro discipline. Recording *declines* ("pre-commit hook
  considered, declined — checkpoint speed") is the rare move that stops a team arguing the
  same thing forever. This is the soul of the kit; everything else is downstream of it.
- **Scar-driven checks that generic linters don't have.** The EOF sentinel (truncation
  hiding inside a trailing comment passes `bash -n` — so require every script to end in
  `exit`/`# EOF`) is genuinely clever and came from a real incident. Same for the NUL-byte
  zero-fill scan and the "0 files checked" vacuous-pass guard.
- **`sync-repo` as a state classifier.** "Fast-forward is the only conflict-impossible
  auto-action; everything else STOPs with plain-language instructions" is exactly the right
  model for multi-machine work, and exactly the git rabbit-hole a beginner must be kept out
  of. This is the most valuable single tool for the target audience.
- **The privacy backstop reports indexes, never marker text.** Output stays shareable even
  while scanning for secrets. Thoughtful.
- **Pure-core / effectful-shell + one-way dependencies + the annotation standard.** This is
  a correct, teachable mental model of software architecture, and the ~120-line exemplar is
  the right way to anchor it. (See §5 — this is also the untapped teaching asset.)
- **Rule of two** against speculative extraction, and the anti-goals section in `ROADMAP.md`
  ("no kit-polishing as procrastination") — self-awareness that most kits lack.

Be careful *not* to "simplify" any of these away in the name of beginner-friendliness. The
fix is disclosure order, not deletion.

## 3. Concrete defects found

These are real, specific, and fixable. The twin-parity pass is the source for most.

### 3.1 SIGNIFICANT — `sync-repo.ps1` is missing the synced-mount guard that `sync-repo.sh` has
`sync-repo.sh` (lines ~32–36) refuses to run when `$PWD` matches `/sessions/*/mnt/*`,
`/mnt/c/*`, `/mnt/d/*` — enforcing the kit's own "never run git through a synced/emulated
mount" rule. **The `.ps1` twin has no equivalent guard** and will happily `git fetch` +
`git merge --ff-only` from a mapped drive, a OneDrive overlay, or a UNC path. This is the
Windows half silently violating a rule the Linux half enforces — and Windows is the primary
platform. **Fix:** add a path check at the top of `sync-repo.ps1` (mapped-drive letters
that resolve to network/overlay, OneDrive-overlay attributes, UNC `\\` prefixes) that STOPs
with exit 2, mirroring the `.sh`. This is the headline bug; it defeats the kit's core
promise on its main OS.

### 3.2 MINOR (privacy-relevant) — `check-kit-docs` marker scan diverges by case and binary handling
The `.ps1` matches case-insensitively and scans binaries under ~1 MB; the `.sh` uses
`grep -rliIF` — **case-sensitive** and `-I` **skips binary files**. A marker differing only
in case, or embedded in something grep treats as binary, is caught on Windows and missed on
Linux. For a privacy backstop, align them (`grep -i`, drop `-I`, or explicitly document the
asymmetry as accepted).

### 3.3 MINOR — `verify-file-integrity.sh` only probes `python3`
The `.ps1` probes `python`/`py`/`python3` and confirms the launcher actually runs; the `.sh`
only tries `python3`. On a Linux/macOS box with `python` but no `python3`, the `.sh`
silently *skips* the Python truncation gate — a vacuous pass exactly like the one the kit
already wrote a guard against.

### 3.4 MINOR — `check-reference-coverage` separator-regex divergence
`.ps1` skips `---*` (≥3 dashes); `.sh` skips `----*` (≥4). A 3-dash, colon-less table
separator passes on Windows but FAILs "unknown status" on Linux. Also the "(with warnings)"
suffix on a passing run exists only on the `.sh`. Exit-code-neutral except the separator edge.

### 3.5 MINOR — `launch-claude-clean.ps1` doesn't run the health gate before launching
The `.sh` runs `verify-path-health.sh` as a hard gate (refuses to launch on FAIL). The
`.ps1` does its own inline PATH rebuild but never invokes `verify-path-health.ps1`. Different
safety models; the Windows side can launch into an unverified environment.

### 3.6 Process gap surfaced by the gate run itself
A fresh clone runs the gate with `[WARN] PRIVATE-MARKERS.list has no markers` and
`[WARN] git hooks not installed`. Correct behavior — but to a *beginner* those WARNs read as
"something is broken." See §6: the beginner-mode health output should phrase maintainer-only
guards as optional, or suppress them until the user opts into sharing.

None of these block anything today; the gate is green. But 3.1 in particular undermines the
kit's central claim on Windows and should be fixed before the kit is handed to anyone new.

## 4. The core tension: two audiences wearing one coat

Almost every friction point traces back to this. The kit serves two people:

- **The Operator** (you): multi-machine, public-bound shared kit, fleet of projects,
  collaborators, IP boundaries, history hygiene. *Needs* all the paranoia.
- **The Newcomer** (the gift recipient): one machine, one private project, one good idea,
  zero context. Needs a runway to a first working thing and a gentle reveal of *why* the
  guardrails exist — only as they're encountered.

Today, 100% of the surface is written for the Operator. `GETTING-STARTED.md` claims "no
experience required" but within two screens asks the reader to reason about PowerShell
execution policy, NTFS mounts, GitHub noreply emails, MSYS2, and `PRIVATE-MARKERS.list`.
Stage 3 is an "agent conflict check"; Stage 6 is *lore roleplay*. This is wonderful content
for an enthusiast and a wall for a newcomer with ideas and no patience for ritual.

**The principle to adopt: progressive disclosure. A guardrail should be invisible until it
catches something.** The newcomer should never *read about* the NTFS truncation hazard —
they should simply be prevented from hitting it, and get the one-paragraph explanation only
*if* they do. The lore is a reward for curiosity, not a prerequisite for building.

Everything in §5 and §6 is an application of this one idea.

## 5. Innovations, ranked by leverage on the goal

### A. Two front doors (highest leverage, lowest effort)
Split the entry experience in two and route at the very top of `README.md`:
- **"I have an idea — let's build."** → a new `START-BUILDING.md`: install (one command,
  see C) → scaffold (interactive, see D) → build with the agent → `gate` → done. Five steps,
  no lore, no privacy apparatus, no fleet management. Every guardrail runs *silently* in the
  background; each only speaks up when it fires, and then in plain language.
- **"I maintain the kit / run a fleet."** → today's `GETTING-STARTED.md` + `AGENTS.md` +
  `BOUNDARY.md` + propagation, retitled `MAINTAINING.md`.

This is mostly reorganization of content that already exists. It is the single change that
most moves the kit toward the stated goal.

### B. The agent as tutor — "Explain mode" (this is what delivers "learn how code works")
The annotation standard (`@intent/@flow/@boundary/@invariant/@risk`) is already a *complete
curriculum* in how good code is structured — it's just buried as machine-readable metadata.
Surface it as teaching. Add a skill (`explain-mode` / `learning-mode`) that instructs the
agent, whenever it generates or edits a module, to narrate in plain English: *what this file
is for, why it lives where it does, why the math is exact, why nothing here touches the disk,
why the test mirrors it.* The beginner watches the architecture get explained as it's built,
not handed to them as rules. The conventions stop being "rules the agent obeys" and become
"the lesson the human absorbs." This is the mechanism by which a finished first project turns
into the promised "zoom out and understand programming."

### C. One-command bootstrap (collapses Stage 0–3)
Replace six stages of copy-paste with a single guided launcher (`bootstrap.ps1`/`.sh`, a
twin pair like everything else) that: checks/install-prompts git + Python, sets a noreply
identity if unset, clones to a safe path (refusing cloud-synced/NTFS locations *for* them),
runs `verify-path-health` + `gate`, installs hooks, and prints one friendly "you're ready"
or one specific fix. The kit is fanatically cross-platform in its tools but the *setup* is
manual ritual; this closes that gap and is the beginner's actual first impression.

### D. Build the interactive `-Features` scaffolder (already designed, not built)
`APP-ARCHITECT.md` §6 and `ROADMAP.md` WS2 already specify a feature-driven scaffolder; it's
the highest-impact unbuilt item for this audience. A beginner can't compose presets in their
head. Make `new-project` (with no `-Features`) drop into a plain-language wizard: *"Should it
remember things after you close it? Will you use it on your phone too? Does it handle money?"*
— each answer adds or removes a module, then the agent generates the annotated skeleton that
gates green before any feature code. This is the "surprising speed" lever made real.

### E. Symmetric safety — add `rescue` to match `checkpoint`
The kit is superb at *save* (`checkpoint`) and *detect* (integrity/identity) but a beginner
who actually tangles a repo (diverged + conflict) is told to run git incantations they can't.
Add a `rescue`/`undo` twin: snapshot the current state to a `rescue/<timestamp>` branch, then
return the working tree to last-known-good — a guided "get me back to safe" that never
destroys work. `checkpoint` saves; `rescue` restores. For this audience, "oh no, fix it" is
as important as "save it."

### F. Frictionless feedback — `report-snag`
The learning loop's *input* currently requires the user to hand-write a `LESSONS.md` entry —
a beginner never will. `GETTING-STARTED` even says "the error message IS a contribution" but
offers no tool, just "send it to whoever invited you." Add a `report-snag` command that
captures the last command, its output, and sanitized environment into a pre-formatted
`LESSONS.md` draft (and offers to copy/share it). This makes the newcomer the "most valuable
test the kit has ever had" *mechanically*, not just aspirationally — and feeds your retro
loop with real first-contact friction.

### G. The capstone — a generated "what you learned" retro
To actually *deliver* the insight payoff, end a project (or milestone) with an agent-generated
retrospective that maps what the person built onto the universal concepts: "you used a *pure
core* (here), *durable writes* (here), a *one-way dependency boundary* (here), *tests that
mirror code* (here) — these are the same ideas behind every serious program." Make the
zoom-out explicit instead of hoping it's absorbed. This is the difference between "I made a
thing" and "I understand how things are made."

### H. macOS support (reach-limiter for a *gift*)
macOS is explicitly unsupported ("scripts need GNU tools"). The person-with-ideas you want to
hand this to is at least as likely to be on a Mac as on Windows. The `.sh` twins are close;
the blockers are a handful of GNU-isms (`stat -f -c`, `sed -i` syntax, `grep -P`). Either
audit the twins for BSD/coreutils portability (or `brew install coreutils` in bootstrap), or
state the limitation loudly up front. As written, the kit's reach for its own goal is cut by
the largest segment of creative non-programmers.

## 6. "Less thorough" — what to hide or defer (you asked specifically)

The kit is not too thorough *in capability*; it's too thorough *in what it shows the
newcomer up front*. Demote the following out of the beginner critical path (they stay fully
present for the Operator):

- **Stage 3 (agent-conflict check)** and **Stage 6 (lore roleplay)** → move to an "Going
  deeper" appendix. Brilliant for enthusiasts, a wall for newcomers.
- **The entire privacy apparatus** (`BOUNDARY.md`, `PRIVATE-MARKERS.list`, the content +
  identity + history scans, install-hooks) → this solves the *Operator's* problem of not
  leaking into a *public shared kit*. A beginner building a private app doesn't have it yet.
  Keep it; just don't make it part of "here's how you start." In beginner mode, the
  session-start health output should not emit `markers missing` / `hooks not installed`
  WARNs — phrase them as "(optional — set up when you're ready to share this publicly)" or
  suppress until opt-in. Right now they read as breakage (see §3.6).
- **`propagate-tools`, `KIT-VERSION`, `TARGETS.list`, fleet management** → these only matter
  once a *second* project exists. By the kit's own rule of two, they shouldn't be on the
  newcomer's first page at all.
- **`repo-study` ("student mode")** → an Operator/learning-loop tool; not a first-project
  concern.

Net: the beginner's mental model should be exactly six nouns — *idea, scaffold, plan, build,
gate, status* — and nothing else until they reach for it.

One genuine over-build to reconsider on the merits (not just hide): the doc set has grown to
~15 top-level/maintained markdown files. For the Operator that's fine; for navigation it's
worth a one-screen "which doc, when, for whom" map at the top of the README so no reader has
to hold all of them in their head.

## 7. One careful re-raise (not a re-litigation)

`LESSONS.md` records that a **pre-commit hook running the full or cheap gate was declined**,
because it would block fast `wip:` checkpoint commits — a sound call, and I'm not reopening
it. But the decision evaluated "full" and "cheap-checks-only"; it did **not** evaluate
*"cheap-checks-only that auto-exempts `wip:` commits by inspecting the commit message."*
That variant threads the exact needle the decline was protecting: a beginner (who will never
remember to run `gate`) gets mechanical enforcement on *real* commits, while `checkpoint`'s
`wip:` panic-saves pass straight through untouched. The existing `pre-commit` hook is already
the right home and already skips cleanly when prerequisites are absent. Worth a fresh entry
as a *new option*, decided on its own merits — especially because honor-system gating fails
precisely for the audience this kit now wants to serve.

## 8. Suggested sequence

1. **Fix `sync-repo.ps1`'s missing mount guard (§3.1).** Credibility + the core promise on
   Windows. Do this first.
2. **Two front doors (§5A) + demote maintainer content (§6).** Pure reorg; transforms the
   first impression.
3. **One-command bootstrap (§5C).** The newcomer's real first five minutes.
4. **Interactive `-Features` scaffolder (§5D).** The speed payoff; already designed.
5. **Explain mode + the capstone retro (§5B, §5G).** The *learning* payoff — the whole point.
6. **`rescue` (§5E) and `report-snag` (§5F).** Safety net + feedback loop for real beginners.
7. **Clean up the minor twin drift (§3.2–3.5); decide macOS (§5H); log the pre-commit
   re-raise (§7).**

Items 1–4 make a non-programmer productive fast; 5 makes them *understand*; 6–7 harden it for
real-world first contact. None of it requires weakening a single existing guardrail — only
reordering when each one speaks.
